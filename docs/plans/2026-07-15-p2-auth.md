## Goal
Stand up authentication, session, and authorization for the FastAPI backend using
**FastAPI-Users**, preserving the migrated argon2id logins (no forced reset) and reproducing the
Flask deny-by-default gate + admin gate + password rules, with password-reset and basic OAuth as
first-class future flows.

## Context
This supersedes the "minimal custom" lean in `2026-07-14-porting-spec.md` (section 3). Adrian
chose the standard batteries-included stack (FastAPI-Users) so password reset + basic OAuth ship
built-in and we avoid a future auth re-platform. Verified against current FastAPI-Users docs
(context7, HIGH confidence): integer-PK user table, custom `PasswordHelperProtocol`, cookie
transport + database strategy (opaque HttpOnly session, not JWT), reset-password + OAuth routers,
and the `validate_password` hook. P2a hash probe (done): passwords are
`argon2id(base64(HMAC_SHA512(SECURITY_PASSWORD_SALT, password)))`, argon2 params m=65536/t=3/p=4.

## Approach
Add `fastapi-users[sqlalchemy]` (+ `httpx-oauth` for OAuth, wired later).

- **User model** = `SQLAlchemyBaseUserTable[int]`, reconciled to our migrated fsqla schema by
  ALIASING FastAPI-Users' attributes onto the existing DB columns (no data rename):
  - `hashed_password` -> column `password`
  - `is_active` -> column `active`
  - `is_superuser` -> column `is_admin` (superuser == our admin; `require_superuser` covers the
    admin gate)
  - keep `name`, `fs_uniquifier`, and every other fsqla column as-is
  - ADD an `is_verified` column (additive Alembic migration; FastAPI-Users requires the
    attribute; also future-proofs email verification). Non-confirmable now -> default True.
  - Build-time check: overriding an inherited mapped column from `SQLAlchemyBaseUserTable` (to
    point at an existing DB column name) must be verified against the installed version's
    declaration style; fall back to explicit `__table_args__`/`synonym` if needed.

- **Custom PasswordHelper** implementing the exact scheme
  `argon2id(base64(HMAC_SHA512(SALT, password)))` (m=65536/t=3/p=4). `verify_and_update` returns
  `(verified, None)` ALWAYS -> disables FastAPI-Users' auto-rehash so migrated hashes are never
  silently rewritten. `hash()` applies the same construction for new passwords. Uses the real
  `SECURITY_PASSWORD_SALT` (dev dummy now; real salt only at cutover, per P8).

- **Session** = `CookieTransport(cookie_httponly=True, cookie_samesite="lax", cookie_secure` in
  prod`)` + `DatabaseStrategy` (opaque token in a new `access_token` table in Postgres -> DB stays
  the source of truth, server-side revocation). Redis strategy is a trivial swap if we later want
  session reads off the DB. Signed/managed by `SECRET_KEY`.

- **Deny-by-default** = a FastAPI middleware allowlisting ONLY the auth routes + `/health` +
  OpenAPI docs; everything else requires an authenticated user. Content-negotiate 401-JSON vs
  302-`/login`. (FastAPI-Users gives per-route deps; we ADD the global middleware to keep the
  Flask guarantee that a newly added route is protected the moment it is registered.)

- **Admin gate**: paths under `/api/admin` (and `/admin` once the frontend lands) require
  `is_superuser` (== `is_admin`) -> 403.

- **Password rule**: `UserManager.validate_password` reproduces 8+ chars / >=1 digit / >=1 symbol;
  required `name` on the `UserCreate` schema (server-side twin of the UI rule).

- **CSRF**: `SameSite=Lax` on the session cookie is the protection (blocks cross-site
  POST/PUT/DELETE from carrying the cookie; all our state changes are unsafe methods, not GET). No
  double-submit token -- redundant for a same-origin, LAN-only app (decided 2026-07-15; CSRF is
  fired by the victim's browser, so "LAN-only" is not itself the mitigation, but SameSite is).
  Lax (not Strict) so the future OAuth redirect-back still carries the session. Revisit at the P8
  HIPAA gate if the app ever becomes internet-facing.

- **Admin bootstrap**: a console script `python -m app.cli admin grant|revoke|list <email>` on a
  session, mirroring the Flask `cli.py`.

### Alternatives rejected
- **Minimal custom cookie+argon2** (prior lean): would re-implement password reset + OAuth by
  hand later; Adrian wants the standard stack.
- **JWT strategy**: opaque DB session is better for PHI (server-side revocation, no token in JS).
- **Renaming DB columns to FastAPI-Users defaults**: breaks the 1:1 P1b migration; aliasing
  preserves the migrated rows.

## Deviations from P1 locked decisions (need ratification)
1. **Session identity is no longer `fs_uniquifier`.** FastAPI-Users keys sessions by user id via
   its `access_token` table. Logins still survive (preserved hash + custom PasswordHelper is the
   real guarantee); `fs_uniquifier` stays as a column but is no longer the session key.
2. **Two additive schema changes** on top of the P1b baseline `73abdcd5ef01`: a new `access_token`
   table + a new `is_verified` column on `user`. Existing rows keep `password`/`active`/`is_admin`.

## Risk / Rollback
- Blast radius: only `feat/nextjs-fastapi-rewrite`; `main` (Flask) is unaffected.
- Rollback: revert the P2 commits; P1 stands (schema baseline + migrated data intact).
- Top risks: (a) inherited-mapped-column override pattern -> verify at build, fall back to
  `synonym`; (b) auto-rehash silently rewriting migrated hashes -> neutralized by
  `verify_and_update` returning `None` (a test asserts it); (c) the real salt is absent in dev ->
  test the verifier with a dev-salt user, real salt only at cutover.
