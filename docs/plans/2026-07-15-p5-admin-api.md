## Goal
Port the 7-route `/api/admin` surface (category + prompt administration + admin reprocess) from the
Flask `admin_api` blueprint to a FastAPI router, preserving the immutable-id/soft-delete rules, the
revision-bump-on-edit (which invalidates the worker classifier caches), and the admin-scoped
reprocess that reuses the P4 summarize enqueue.

## Context
Porting spec section 1 lists `admin_api` (7 routes, is_admin-gated) as keep/rewrite. The catalog +
audit services are already ported (P3a, session-based); `catalog.bump_revision(session)` already
exists; the summarize enqueue exists (P4). So P5 is a thin HTTP layer, mirroring the P3 documents
router.

## Locked decisions (confirmed with Adrian, 2026-07-15)
- **Admin auth = app-level gate + a per-route superuser dependency** (defense-in-depth): keep the
  P2b `enforce_auth` 403 on `/api/admin/*`, AND put `Depends(current_superuser)` on the admin
  router so the surface stays protected even if the allowlist is ever mis-edited (and is testable in
  isolation).
- **No hard delete**: category ids are immutable (they key stored review rows); deactivation is a
  soft-delete via `PATCH active=False`. Mirror the Flask design - no DELETE route.
- **reprocess bypasses owner-scoping** (admin acts on ANY owner's document) - resolve by id (404),
  require an included row (400), reuse `jobs.enqueue(... "summarize" ...)` -> JobConflict -> 409.
  Preserve this asymmetry vs the owner-scoped documents routes (porting spec is explicit).
- **Revision bump = the classifier reload**: every category/prompt edit calls
  `catalog.bump_revision(session)`; the worker classifier caches self-invalidate on their next
  `classify()` (they poll `catalog_version()`), so no cross-process reload signal is needed (the P4
  worker design already handles it).
- **One batch** (small, mechanical).

## Approach
- `app/schemas/admin.py`: `CategoryCreate` (id numeric-string, name required, examples list,
  active/auto_assign), `CategoryUpdate` (all optional; id NOT accepted), `PromptPut` (text).
- `app/api/admin.py`: `APIRouter(prefix="/api/admin", dependencies=[Depends(current_superuser)])`
  with the 7 routes. `_category_payload` adds `has_summary_prompt`. All DB access on the sync
  `get_db` session; edits call `bump_revision` + `audit`. Wire into `main.py`.
- reprocess resolves the document by id (no owner filter), checks included rows, calls
  `jobs.enqueue`, maps `JobConflict` -> 409.

### Alternatives rejected
- App-level gate only (Flask parity): simpler but the admin protection depends solely on the
  allowlist; Adrian chose defense-in-depth.
- A hard-delete route: would orphan stored review rows referencing the id; soft-delete preserves
  history.

## Tasks
- **P5 - admin router.** approach: test-after.
  - files: app/schemas/admin.py, app/api/admin.py, app/main.py, backend/tests/test_admin_api.py.
  - acceptance: a non-admin authenticated user gets 403 on every /api/admin route (the app gate +
    the router dep); whoami returns the admin's email + is_admin; create category (valid + rejects
    bad id / duplicate / missing name); update category (soft-delete via active=False; id immutable);
    get/put summary prompt (put bumps revision); reprocess enqueues for any owner (404 unknown doc,
    400 no included rows, 409 when a job is active); every edit writes an audit row + bumps the
    catalog revision.

## Risk / Rollback
- Blast radius: only `feat/nextjs-fastapi-rewrite`; `main` unaffected.
- Rollback: revert the P5 commit; P1-P4 stand.
- Top risks: (a) the admin router dep double-resolving the user (minor cost, acceptable); (b) a
  category-id validation gap letting a non-numeric id through -> mirror the Flask `_ID_RE` check;
  (c) reprocess acting cross-owner is INTENTIONAL - a test asserts it (admin reprocesses another
  user's document).
